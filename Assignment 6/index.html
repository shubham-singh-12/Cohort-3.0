Attributes vs Properties

Understand the key difference between HTML attributes and DOM properties.

🔬 Live Demonstration

Type something in the input, then click Check Difference to see how input.value (DOM Property) differs from input.getAttribute("value") (HTML Attribute).

input.value (Property)
input.value
—
Reflects the current live value — changes as user types.
getAttribute("value") (Attribute)
input.getAttribute("value")
—
Reflects the original HTML value — stays as set in HTML.
// ── HTML in the file ───────────────────────────
<input type="text" id="attrDemoInput" value="initial value">
 
// ── After user types "hello" ────────────────────
const input = document.getElementById("attrDemoInput");
 
// PROPERTY: reflects current live state of DOM node
input.value // → "hello" ← what user typed
 
// ATTRIBUTE: reflects the original HTML markup value
input.getAttribute("value") // → "initial value" ← from HTML
 
// KEY INSIGHT:
// Attribute = source of truth at parse time
// Property = live current state of the element
// They start equal; diverge once user interacts.

card.removeAttribute("data-status")

Event Propagation

Visualize Bubbling, Capturing, and Event Delegation in action.

🫧 Event Bubbling

Events travel bottom → up: Child fires first, then Parent, then Grandparent. Default behavior.

Parent

🎯 Event Capturing

Events travel top → down: Grandparent fires first, then Parent, then Child. Use { capture: true }.

Grandparent
Parent
// Click the button to see capturing order…
// ── EVENT BUBBLING (default) ──────────────────────
// Listener added with capture: false (default)
 
grandparent.addEventListener("click", () => console.log("Grandparent")) // fires 3rd
parent.addEventListener("click", () => console.log("Parent")) // fires 2nd
child.addEventListener("click", () => console.log("Child")) // fires 1st
 
// Output order → Child → Parent → Grandparent
 
// ── EVENT CAPTURING ───────────────────────────────
// Listener added with { capture: true }
 
grandparent.addEventListener("click", () => console.log("Grandparent"), { capture: true }) // fires 1st
parent.addEventListener("click", () => console.log("Parent"), { capture: true }) // fires 2nd
child.addEventListener("click", () => console.log("Child"), { capture: true }) // fires 3rd
 
// Output order → Grandparent → Parent → Child
 
// ── STOP PROPAGATION ──────────────────────────────
child.addEventListener("click", (e) => {
  e.stopPropagation() // prevents event reaching parent/grandparent
})
Event Delegation Key Concept

🎪 How Task Manager Uses Delegation

Instead of attaching 3 listeners (edit, complete, delete) to each task card — which would mean N×3 listeners — we attach ONE listener to the parent container and check e.target.dataset.action.

// ── WITHOUT delegation (bad — N×3 listeners) ─────
tasks.forEach(task => {
  task.querySelector(".edit").addEventListener("click", handleEdit)
  task.querySelector(".complete").addEventListener("click", handleComplete)
  task.querySelector(".delete").addEventListener("click", handleDelete)
})
 
// ── WITH delegation (good — 1 listener total) ─────
document.getElementById("task-container").addEventListener("click", (e) => {
  const action = e.target.closest("[data-action]")?.dataset.action
  const card = e.target.closest(".task-card")
  if (!card) return
 
  if (action === "delete") deleteTask(card)
  if (action === "complete") completeTask(card)
  if (action === "edit") editTask(card)
})
 
// Works for ALL tasks — even ones added AFTER the listener!

Browser Rendering Pipeline

How the browser converts HTML + CSS into pixels on screen.

📄 HTML → DOM Tree

HTML
Raw .html file received from server
↓
Parsing
Browser reads bytes → characters → input stream
↓
Tokenization
Characters split into tokens: StartTag, EndTag, Text, Comment
↓
DOM Tree
Tokens → Node objects → parent-child tree structure

🎨 CSS → CSSOM Tree

CSS
Raw .css file or <style> block received
↓
Parsing
CSS bytes → characters, applying cascade rules
↓
Tokenization
Selectors, properties, values tokenized
↓
CSSOM Tree
Computed styles for each node — mirrors DOM structure

🖼 Combining → Render Tree → Pixels

DOM Tree
Nodes & structure
+
CSSOM Tree
Styles & rules
→
Render Tree
Only visible nodes with computed styles
↓
Layout (Reflow)
Browser calculates exact position & size of every box (Box Model)
↓
Paint
Browser fills in pixels: colors, borders, shadows, text
↓
Compositing
Layers assembled and sent to GPU → displayed on screen ✓
📝
Parsing
The browser reads the raw byte stream and changes it into a useful structure. The HTML parser is error-resistant and manages incorrect markup.
🔤
Tokenization
The input stream is divided into discrete tokens: StartTag(<div>) EndTag(</div>) Character Comment DOCTYPE Each token has attributes and name.
🌳
DOM Tree
These tokens get represented as Node objects which are arranged in the form of a tree. Each HTML element becomes a node, having parent-child and sibling relationship.
🎨
CSSOM Tree
This CSS gets parsed to become another tree similar to the previous one, having information about styles mapped on to all DOM nodes.
🖼
Render Tree
DOM + CSSOM together. Only visible nodes included - display:none nodes are excluded. Each node has its geometry and style.
⚡
Reflow vs Repaint
Layout (width, height) changes cause Reflow - costly. Changing color/opacity will trigger the repaint only. Use transform to animate.
